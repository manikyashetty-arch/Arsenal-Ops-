# Hierarchy UX — Single PR Scope

Branch base: `ui-redesign`.
Goal: maximize hierarchy UI/UX in one PR; touch backend zero times.

## In scope (4 capabilities, one PR)

1. **Kanban card chips** — `EpicChip` (inline above title) + `ParentChip` (corner) on every card with `epic_id` / `parent_id`. Click → open referenced item in detail panel.
2. **List view: Group by Epic toggle** — extend existing List view with `Group by: Sprint · Epic` toggle. Epic mode groups items under their epic, indents subtasks under parent task, "Unparented" section at bottom. Collapse state persisted to localStorage per project.
3. **Detail panel: parent/epic pickers** — `EpicPicker` (for stories/tasks/bugs) and `ParentPicker` (for tasks) in detail panel edit form, between Assignee and Sprint. Searchable Combobox over cmdk. Type-pair + cycle filtering. Clear option to unset.
4. **Optimistic UI + client-side validation** — `validateReparent` (type-pair + cycle rules), `useOptimisticReparent` (React 19 `useOptimistic` + manual rollback + toast).

## Out of scope (deferred to follow-up PRs)

- Brand-new HierarchyView tab.
- Multi-select + bulk reparent.
- Ghost rows for cross-sprint kin.
- Full APG treegrid keyboard contract.
- `TicketDetailPanel.tsx` (my-tasks panel) edit affordance.
- Backend type-pair + cycle validators.
- Frontend test infrastructure (Vitest + RTL).
- `ProjectBoard.tsx` carve-up.

## Type-pair rules (client-enforced)

| Subject | `epic_id` allowed | `parent_id` allowed |
|---|---|---|
| epic | — | — |
| user_story | epic | — |
| task | epic | task or user_story |
| bug | epic | — |

Self and cycle-creating targets always excluded.

## Acceptance criteria

- Every kanban card with `epic_id` shows the epic chip; clicking opens that epic.
- Subtask cards show corner parent chip; clicking opens parent.
- List view has `Group by Sprint | Epic` toggle. Default: Sprint (existing).
- In Epic mode: items grouped under epic; subtasks indented under parent task with rail; Unparented section last.
- Collapse state persists in localStorage per project.
- Detail-panel edit form has Epic picker (story/task/bug) and Parent picker (task).
- Pickers exclude self and cycle targets; type-pair rules enforced; "Clear" option works.
- Reparent updates optimistically; on PUT failure, UI reverts + toast.
- All existing kanban/list/detail-panel behavior unchanged when hierarchy fields aren't being touched.

## Notes

- API already returns `parent_key` / `epic_key` and accepts `parent_id` / `epic_id` in PUT `/api/workitems/{id}` — no backend change needed.
- Existing `handleSaveEdit` sends full `editForm` JSON — adding the new fields requires no edit to save logic.
- Existing `selectedItem` carries `epic_id` / `parent_id` so `setEditForm(selectedItem)` (called when entering edit mode) seeds the pickers correctly.
